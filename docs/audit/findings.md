## หมายเหตุ INV coverage (เส้นทางที่แตะข้อมูลแต่ไม่รักษา INV)

- **INV-07/28 (payment↔record referential):** เส้นทางที่ลบ record มี 2 — API `[id]/route.js:292` และ client `records/page.js:908` — **ทั้งสองไม่มี guard** → #6.
- **INV-09/10 (atomicity):** เส้นทาง payment มี 4 (create API/client, cancel API/client). ยืนยันฝั่ง API ไม่ atomic (#2,#3). Client dual (`create/page.js`, `payments/page.js`) มีรูปแบบเดียวกัน (§3) → เสี่ยงเท่ากัน แต่ไม่รักษา INV เพิ่ม.
- **INV-30 (period-level dup) — ใหม่:** แม้ #2 ปิด atomicity gap + ปิด INV-08 ระดับ record ใน tx แล้ว (`payments/route.js:180-182` เช็ค `isPaid`) แต่ dup-guard ระดับ **period** ยังอยู่นอก tx (`:120-125`) + doc id random (`:166`) → concurrent 2 request (userId+period เดียว, recordIds ไม่ทับ) ยังสร้าง payment ซ้ำได้ → #20. ความเสี่ยงต่ำ (admin-only, window แคบ, ผลลัพธ์มองเห็น+cancel ได้) แต่ปิดสนิทต้องเปลี่ยนเป็น deterministic doc id (นอก scope #2/#3).
- **INV-13/27 (Storage lifecycle):** เส้นทางลบที่รักษา = fishingRecords (`[id]/route.js:261-292`). ไม่รักษา = wisdom (#7); news image และ fish-species image — data-flow ระบุ "ต้องยืนยัน ถ้าลบได้" (§5.3 ข้อ 3-4) ยังไม่ยืนยันว่ามี path ลบจริง → คงเป็น candidate ไม่ยกเป็น finding แยก.
- **INV-24 (role-less→ADMIN):** เส้นทางเดียวที่ตั้ง role อัตโนมัติคือ `AuthContext.js:41-48` (existing doc ไม่มี role) และ `:59-84` (ไม่มี doc → default RESEARCHER/ADMIN ตาม email). เส้นแรก = ช่องโหว่ (#1); เส้นที่สอง default เป็น RESEARCHER สำหรับ email ทั่วไป (ความเสี่ยงต่ำกว่า แต่ยัง auto-provision — ควรทบทวน).

## ข้อที่ยัง "ต้องยืนยัน" (ไม่ยกเป็น finding เพราะ mobile origin / นอก repo)

- INV-02/03/26 — `fisherName`/`recordedBy`/`fisherInfo`/`localName` persist โดย mobile app (คนละ repo) — ฝั่ง web ไม่มีจุด write ให้ตรวจ.
- INV-08 — atomic guard กันเลือก record ซ้ำแบบ concurrent — create อ่าน `availableRecords` แต่ไม่มี transaction lock (เกี่ยวกับ #2).
